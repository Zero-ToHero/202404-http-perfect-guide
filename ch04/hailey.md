# 커넥션 관리
## TCP 커넥션
- HTTP 에게 신뢰할 만한 통신 방식을 제공
- IP 패킷이라고 불리는 작은 조각을 통해 데이터를 전송

### TCP 스트림은 세크먼트로 나뉘어 IP 패킷을 통해 전송 됨
1. 현재 연결되어있는 TCP 커넥션을 통해 메시지 데이터의 내용을 순서대로 보냄
2. TCP 는 세크먼트라는 단위로 데이터 스트림을 잘게 나눔
3. 세그먼트를 IP 패킷이라고 불리는 봉투에 담아서 인터넷을 통해 데이터를 전달
4. IP 패킷 구성
   - IP 패킷 헤더 : 발신지와 목적지 IP 주소, 크기, 기타 플래그
   - TCP 세크먼트 헤더 : TCP 포트 번호, TCP 제어 플래그, 데이터의 순서와 무결성을 검사하기 위해 사용되는 숫자 값을 포함
   - TCP 데이터 조각

### TCP 커넥션 유지
- 컴퓨터는 항상 TCP 커넥션을 여러 개 가지고 있음
- TCP 포트 번호를 통해 여러 개의 커넥션을 유지
- IP 주소는 해당 컴퓨터에 연결 되고 포트 번호는 애플리케이션으로 연결 됨
- TCP 커넥션 식별 값은 4가지 값으로 이루어져 있고 해당 값으로 유일한 커넥션을 생성 함
```
<발신지 IP 주소, 발신지 포트, 수신지 IP 주소, 수신지 포트>
```

### TCP 소켓 프로그래밍
- 운영체제는 TCP 커넥션의 생성과 관련된 여러 기능을 제공
- HTTP 트랜잭션을 수행하기 위한 소켓 API 의 사용 법
1. 웹서버는 커넥션을 기다림
2. 클라이언트는 URL 에서 IP 주소, 포트 번호를 알아내고 TCP 커넥션을 생성하기 시작
   1. 커넥션 생성은 서버와의 거리, 서버의 부하, 인터넷 혼잡도에 따라서 시간이 걸림
3. 커넥션이 맺어지면 클라이언트는 HTTP 요청을 보냄
4. 서버에서는 요청을 받고 분석하여 클라이언트가 원하는 동작을 수행한 후 클라이언트에게 데이터를 보냄
5. 클라이언트는 응답 데이터를 처리

## TCP 성능에 대한 고려
- HTTP는 TCP 바로 위에 있는 계층이기 때문에 HTTP 트랜잭션의 성능은 그 아래 TCP 성능에 영향을 받음
### HTTP 트랜잭션지연
1. 최근에 방문한 적이 없는 호스트에 방문하는 경우 DNS 이름 분석 기술을 사용하여 URI에 있는 호스트명을 IP 주소로 변환하는데 수십 초의 시간이 걸림
2. TCP 커넥션 요청을 서버에게 보내고 서버가 커넥션 허가 응답을 회신하기를 기다리는데 커넥션 설정 시간은 새로운 TCP 커넥션에서 항상 발생 함
3. 요청 메시지가 인터넷을 통해 전달되고 서버에 의해서 처리되는데까지 소요된느 시간
4. 웹 서버가 HTTP 응답을 보내는 시간
### 성능 관련 중요 요소
1. TCP 커넥션 핸드셰이크 지연
- 새로운 TCP 커넥션을 열 때 TCP 소프트웨어는 커넥션을 맺기 위한 조건을 맞추기 위해 연속으로 IP 패킷을 교환하는데 작은 크기의 데이터 전송에 커넥션이 사용된다면 HTTP 성능을 크게 저하시킴
- TCP의 ACK 패킷은 HTTP 요청 메시지 전체를 전달할 수 있을 만큼 큰 경우가 많고, 많은 HTTP 서버 응답 메시지는 하나의 IP 패킷에도 담길 수 있기 때문에 크기가 작은 HTTP 트랜잭션은 SYN/SYN+ACK 핸드셰이크가 눈에 띄는 지연을 발생시킴
2. 확인응답 지연
- 인터넷 자체가 패킷 전송을 완벽히 보장하지 않기 때문에 성공적인 데이터 전송을 보장하기 위해 자체적인 확인 체계를 가짐
- 확인응답 지연은 송출할 확인응답을 특정 시간 동안 버퍼에 저장해 두고 확인응답을 편승시키기 위한 송출 데이터 패킷을 찾음
- 요청과 응답 두 가지 형식으로만 이루어지는 HTTP 동작 방식은 확인 응답이 송출 데이터 패킷에 편승할 기회를 감소 시킴 => 지연이 자주 발생
3. TCP 느린 시작
- TCP 의 데이터 전송 속도는 TCP 커넥션이 만들어진 지 얼마나 지났는지에 따라 달라 짐
- TCP 커넥션은 시간이 지나면서 자체적으로 튜닝 되어서 처음에는 커넥션의 최대 속도를 제한하고 데이터가 성공적으로 전송됨에 따라서 속도 제한을 높임
- 인터넷의 급작스러운 부하와 혼잡을 방지하는데 사용
- TCP가 한 번에 전송할 수 있는 패킷의 수를 제한
- 새로운 커넥션은 기존에 존재하는 커넥션 보다 속도가 느림
4. 네이글(Nagle) 알고리즘과 TCP_NODELAY
- 네트워크 효울을 위해 패킷을 전송하기 전에 많은 양의 TCP 데이터를 한 개의 덩어리로 합침
- 크기가 작은 HTTP 메시지는 패킷을 채우지 못하기 때문에 추가적인 데이터를 기다리며 지연 발생
- 확인 응답이 도착할 때까지 데이터 전송을 멈추고 있는데 확인응답 지연 알고리즘은 확인응답을 100~200 밀리초 지연 시킴
- HTTP 애플리케이션은 성능 향상을 위해 HTTP 스택에 TCP_NODELAY 파라미터 값을 설정하여 네이글 알고리즘을 비활성화 시킴
5. TIME_WAIT의 누적과 포트 고갈
- 성능 측정 시에 심각한 성능 저하를 발생시킴
- 같은 주소와 포트 번호를 사용하는 새로운 TCP 커넥션이 일정 시간 동에는 생성되지 않게 하기위해 TCP 커넥션이 끊기면 커넥션의 IP 주소와 포트 번호를 메모리의 작은 제어영역에 기록해 놓음
- 일반적으로 2분정도 동일한 커넥션을 생성하지 못하고 2분의 커넥션 종료 지연이 문제가 되지 않지만 성능 시험 상황에서는 문제가 될 수 있음

## 커넥션 관리
### 흔히 잘 못 이해하는 Connection 헤더
- HTTP Connection 헤더 필드는 커넥션 토큰을 쉼표로 구분하여 가지고 그 값들은 다른 커넥션에 전달되지 않음
- 커넥션 토큰이 HTTP 헤더 필드 명을 가지고 있으면, 해당 필드들은 현재 커넥션만을 위한 정보이므로 다음 커넥션에 전달하면 안 됨
- Connection 헤더에 있는 모든 헤더 필드는 메시지를 다른 곳으로 전달하는 시점에 삭제되어야 함
- Connection 헤더에는 홉별(hop-by-hop) 헤더 명을 기술하는데 이것을 헤더 보호하기 라고 함
### 순차적인 트랜잭션 처리에 의한 지연
- 순차적인 처리로 인해 각 트랜잭션이 새로운 커넥션을 필요로해서 커넥션을 맺는다면 커넥션을 맺는데 발생하는 지연과 함께 느린 시작 지연이 발생 함
1. 병렬 커넥션
- HTTP는 클라이언트가 여러 개의 커넥션을 맺음으로써 여러 개의 HTTP 트랜잭션을 병렬로 처리할 수 있게 함
- 단일 커넥션의 대역폭 제한과 커넥션이 동작하지 않고 있는 시간을 활용하여 각 커넥션의 요청과 전송 시간을 겹치게 하면 커넥션 지연 시간을 겹치게 되면서 총 지연시간이 줄어듬
- 네트워크 대역폭이 좁은 경우 재한된 대역폭 내에서 각 객체를 전송받는 것이기 때문에 성능상의 장점은 거의 없음
- 다수의 커넥션은 메모리를 많이 소모하고 자체적인 성능 문제를 발생시키기 때문에 적은 수의 병렬 커넥션만을 허용
- 화면에 여러 개의 객체가 동시에 보이면서 내려받고 있는 상황을 볼 수 있기 때문에 사용자 입장에서 더 빠른 것처럼 느낄 수 있음
2. 지속 커넥션
- 보통 동일한 사이트에 리소스를 가져오기 때문에 처리가 완료된 후에도 계속 연결된 상태로 있음
- 커넥션을 맺기 위한 사전 작업과 지연을 줄여주고 튜닝된 커넥션을 유지하며 커넥션의 수를 줄여줌
- keep-alive 커넥션
  - 커넥션을 맺고 끊는데 필요한 작없이 없어서 시간이 단축됨
  - HTTP/1.0 keep-alive 커넥션을 구현한 클라이언트는 커넥션을 유지하기 위해서 요청에 Connection:Keep-alive 헤더를 포함시킴
  - 그 다음에도 요청에도 이 커넥션을 통해 받고자 한다면 서버는 응답 메시지에 같은 헤더를 포함시켜 응답 함
  - 해당 헤더가 없으면 클라리언트는 서버가 keep-alive 를 지원하지 않으며 서버 커넥션을 끊을 것이라 추정
  - 옵션
    - timeout : 커넥션을 얼마간 유지될 것인지를 의미하지만 이대로 동작한다는 보장은 없음
    - max : 몇 개의 HTTP 트랜잭션을 처리할 때까지 유지될 것인지를 의미하지만 이대로 동작한다는 보장은 없음
  - 제한과 규칙
    - Connection:Keep-Alive 가 요청 헤더에 있는 경우만 사용 가능
    - 커넥션을 계속유지하기 위해서는 모든 메시지에 Connection:Keep-Alive 헤더를 포함해야 함
    - Connection:Keep-Alive 응답 헤더가 없다면 서버가 응답 후에 커넥션을 끊는 다는 의미
    - 엔티티 본문의 정확한 길이를 알 수 있어야 커넥션을 유지 할 수 잇음
  - 멍청한(dumb) 프락시
    - 프락시는 Connection 헤더를 이해하지 못하기 때문에 다음 서버에 keep-alive 를 그대로 전달
    - 서버와 클라이언트는 keep-alive 를 받았기 때문에 커넥션을 유지하려고 하지만 프락시는 keep-alive 를 모르기 때문에 커넥션이 끊기기를 기다리면서 문제가 발생
    - 프락시와 게이트웨이는 메시지를 전달하거나 캐시에 넣기 전에 Connection 헤더에 명시된 모든 헤더 필드와 Connection 헤더를 제거해야 함
  - Proxy-Connection
    - 모든 헤더를 무조건 전달하는 문제를 해결하기 위해 Proxy-Connection 헤더를 사용
    - 의미 없는 Proxy-Connection 헤더를 Connection 헤더로 바꿔서 전달하면서 원하던 효과를 얻음
- HTTP/1.1에서의 지속 커넥션
  - keep-alive 커넥션을 지원하지 않는 대신 개선된 지속 커넥션을 지원함
  - 지속 커넥션이 기본으로 활성화되어 있음
  - 커넥션을 끊으려면 Connection:close 헤더를 명시해야 함
3. 파이프라인 커넥션
- HTTP/1.1은 지속 커넥션을 통해서 요청을 파이프라이닝 할 수 있음
- keep-alive 커넥션의 성능을 더 높여줌
- 여러 개의 요청은 응답이 도착하기 전까지 큐에 쌓임

## 커넥션 끊기
- 커넥션을 어떻게 끊을 지에 대한 명확한 기준이 없음
1. 마음대로 커넥션 끊기
2. 우아한 커넥션 끊기
- 전체 끊기와 절반 끊기
- TCP 끊기와 리셋 에러
